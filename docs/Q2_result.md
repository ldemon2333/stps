# Q2 实验结果：Phase-Shift（Step B）净收益 — 4 卡（`BW_MAX=9e5`, `D_MAX=16`）

> 本文吞吐用新口径（剔除冷启动）重新跑出：$\text{throughput\_excl\_cold} = \dfrac{N_{\text{completed}}}{T_{\text{steps}} - \overline{T_{\text{cold}}}}$。**p99 沿用原始口径** $p99(\text{completion\_step} - \text{arrival\_step})$，不剔除 STPS 的 cold-start (`start_offset`)，让 phase-shift 自身引入的等待如实计入尾延迟。平均延迟使用 $\text{effective\_delay} = \max(0, \text{end\_step} - \text{arrival\_step} - \text{cold\_start\_ticks})$（excl. cs 列）。原始口径 `throughput / avg_delay` 仍保留，便于和历史 Q2 文档对照。
>
> **范围说明：** 本轮先以 4 卡完整跑通，16 卡版本因 D=16/STPS 内存峰值在 15 GB 主机上跑不完而暂缓（详见 §7 备注）。

## 0. 结论先行

本轮把 phase-shift 工作点放在拥塞档 `BW_MAX=9e5`（demand p75，traffic_calib.md §3）+ `D_MAX=16`，跑 `cards=4`（tasks=800）。**phase-shift 仍是有效的时间错峰模块，但在此拥塞工作点上和原始 `BW_MAX=5e6` 主实验的画风完全相反**：

**1. 收益：拥塞与尾延迟同时下降。** 参考 [Table Q2-2](#table-q2-2)。phase-shift 把 RR / BestFit / DRF / P2C 的 `avg_cong_ratio` 从约 0.26 压到 0（STPS-spatial 压到约 0.16）；p99 从约 130 tick 降到 ~30 tick（−76%），avg_delay (excl. cs) 降幅 80%+。这是 phase-shift 在拥塞工作点最直接的物理收益——把同步打到 cap 的需求摊到时间维。

**2. 代价：跨卡瞬时 CV 反而升高。** 参考 [Table Q2-2](#table-q2-2)。`card_cv` 普遍上涨 12–50%；LIF 上涨 12–22%。原因：BW=9e5 cap 把不同卡的 demand 削平到同一上限，原本的 baseline CV 已经很低（4 卡 ~0.27）；phase 把每张卡的 demand 各自挪开，造成每张卡的 serve window 错位，跨卡瞬时分布反而被拉开。这是 *cap-clipped 工作点下* phase 的次生效应，和 `BW=5e6` 不拥塞档「phase 降 CV」的方向相反。

**3. 吞吐下降 4–12%，剔除冷启动后仍下降 3–11%。** 参考 [Table Q2-2](#table-q2-2) 的 `Throughput cost (excl. cs)` 列。冷启动平均 5–8 tick / 任务（`mean_cold_start`，几乎等于 `mean_start_offset` 减抖动），按新公式从分母里剔除后吞吐损失从 5–12% 收窄到 3–11%，说明 **冷启动本身占了 1–2% 的「吞吐损失」是计量假象**，剩余 3–10% 才是 Step B 的真实成本。

**4. STPS+phase 在 `D=16` 是异常点：尾延迟反而暴涨。** 参考 [Table Q2-2](#table-q2-2) 最后两行：stps-spatial+phase（即 stps）在 4 卡上 p99 从基线 ~130 跳到 220+ tick，avg_delay (excl. cs) 反向恶化 35–55%。这与 Q2 旧文 §9.3 的结论一致——**STPS 模块叠 phase 必须把 `D_MAX` 降到 2 才能拿到 Pareto-better 配置**；`D=16` 不适合 stps。详见 §6 异常分析。

**使用建议：**

- **拥塞 + 延迟敏感** 的场景下 phase-shift 价值最大：能把 `avg_cong_ratio` 压到 0、把 p99 从 ~130 降到 ~30。
- **吞吐敏感** 的场景需要权衡：剔除冷启动后吞吐仍下降 3–10%，加上 `reject_rate_bw` 上升到 18–22%，说明 cap 即使错峰后仍有部分 demand 打不到带宽，被拒绝。
- **STPS 默认 `D_MAX=2`** 而不是 16，否则 phase 在 stps-spatial 已经切过空间后的剩余维度上找不到合适的 offset，把 p99 拖到 220+ tick。详见 §6 异常分析。

观察（拥塞档下 phase 的 Pareto 反向）：

- **CV 不只是均值上升，分布尾部也被整体抬高**：例如 poisson RR 的 `CV mean / median / p95` 从 `0.2670 / 0.2659 / 0.2818` 升到 `0.3572 / 0.3608 / 0.3960`；bursty P2C 的 p95 从 `0.3068` 抬到 `0.4638`。phase 在 cap-clipped 工作点下不只是把均值推高，而是把高 CV 状态出现的概率也一起推高了。
- **`$Std_{CV}$` 揭示 phase 后波动性并不总是同向变化**：RR / P2C 的 `Std_{CV}` 明显增大（如 poisson RR `0.0104 → 0.0309`）；但 BestFit / DRF / STPS-spatial 的 `Std_{CV}` 反而略降，说明它们虽然 CV 均值变高，但 5-seed 之间更集中。
- **拥塞维度全部消化**：非-STPS 基线的 `mean_cong_ratio` 从 ~0.26 降到 0（见 §4）；STPS-spatial 因为本身已经切过空间，phase 后残留 0.16。
- **p99 是关键收益**：4 个非-STPS 基线 phase 后 p99 都降到 ~30 tick（约 baseline 的 1/4.3），avg_delay (excl. cs) 也降 80% 以上。
- **STPS+phase D=16 是 Pareto 反向**：CV 升高、p99 反而从 ~130 跳到 ~220（详见 [§6](#6-stps--phase--d_max16-异常分析)）。
- **`reject_rate_bw` 上升到 0.18–0.22**：非-STPS 基线 phase=on 后，约 1/5 的任务在 `D_MAX=16` 范围内仍找不到 ≤ cap 的 slot 而被拒。

## 6. STPS + Phase + `D_MAX=16` 异常分析

§2/§3 多次提到 **stps-spatial+phase（即默认 STPS 全模型）在 `D=16` 下 p99 反向飙升**。复现旧 Q2 §9.3 的结论，原因可拆为两点：

1. **stps-spatial 自身已经把同步热点做空间拆分**：每个 sync-group 已经按 hotspot 散到多张卡。Phase 拿到的 demand 时序的「同步残量」已经较小，再用 `D=16` 大窗口去搜索的话，phase 会把零散需求推到稳态窗口的尾部，导致部分任务的实际完成 tick 超过 `steps=512` 边界——这部分被 sim 计入「完成」但 `effective_delay` 高达 200+。
2. **phase + spatial-split 联合做了过度去同步**：每张卡的 demand 被切碎到时间序列末端，造成稀疏 + 高方差的负载形状（参考 §3 STPS demand `time_lif` 反升 18–21%）。这种形状对 cap 不敏感（reject_bw=0），但对完成 tick 极敏感，p99 自然爆。

**结论：STPS 端到端默认应使用 `D_MAX=2`**。旧 Q2 §9.3 在 `BW=9e5/D=2` 下 stps 的 CV、tp、p99 同时都比 D=16 更优（CV 0.293 vs 0.401，tp 1.292 vs 1.167，p99 141 vs 223）。本轮 9e5/D=16 数据再一次证实 D=16 不适合 STPS。

非-STPS 基线（RR / BestFit / DRF / P2C）+ phase + D=16 没有这个问题，因为它们的 baseline demand 仍然高度同步，phase 用大窗口才能压平；这一点和 STPS 模块的需求正好相反。

## 7. 16 卡版本备注

16 卡（`cards=16, tasks=3200`）版本本轮未跑完。原因：

- 单个 16 卡 + `D_MAX=16` STPS-phase worker 的 RSS 峰值 4–6 GB；在 15 GB 主机上跑 2 worker 时一旦同步进入 STPS 阶段就接近 OOM。

- `script/q2_run.py` 已加入 `multiprocessing.Pool(maxtasksperchild=1)` + 增量 CSV 写盘 + resume 机制，下次有 32 GB 节点或更长不间断窗口时可以无缝接着跑：

### 7.4.3 三个核心发现

**发现 1 — phase-shift 是「拥塞缓解器」，不是「负载均衡器」**

- 4 个非-STPS 基线在 **A、C** 两档（cap binding 存在）下，`avg_cong_ratio` 都被 phase 压到 0。在 **B**（cap 不 binding）下，base cong 本来就是 0，phase 没有拥塞可缓解。
- 说明 phase 的物理收益是「把不同卡上挤在同一 tick 的 demand 错峰，避免触发 cap」，这件事只在 *cap 是 binding constraint* 时存在。如果 NoC 完全不拥塞，phase 这一步就没有可优化的目标函数。

**发现 2 — `card_cv` 的方向取决于 base 状态**

- **A（拥塞档）**：phase ON 后 `card_cv` 升高 12–48%（除 STPS-spatial 涨更猛到 +48%）。物理含义：base 的低 CV 来自所有卡同步顶到 cap（被强制压平），phase 错峰后每张卡按自身 demand 自由波动，CV 反而升高——base 那个低 CV 其实是「拥塞副作用」，不是真实的均衡。
- **C（轻拥塞档）**：phase ON 后 `card_cv` 下降 6–23%。这一档 base 没被压平到 cap，phase 既错峰拥塞又均衡负载，方向上双赢。
- **B（无拥塞档）**：phase ON 后 `card_cv` 下降 13–26%。基线本来没拥塞，phase 自由地把高峰段 demand 平移到低谷段，纯粹做错峰均衡。
- → **判定**：phase 改善 `card_cv` 的工作点是「cap 几乎不 binding」（B、C 这种），phase 在 cap 紧绷工作点 (A) 上不会改善 `card_cv`，但仍然有拥塞收益。**`card_cv` 这个指标在 A 档不能用来评价 phase**。

**发现 3 — `p99` 的方向相反，取决于「base 是否被拥塞拖延」**

- **A（拥塞档）**：base p99 ≈ 125–133（被 NoC 排队拖到 100+ tick），phase 把 p99 压到 ~16（剔除冷启动后基本退化为服务时间）。p99 是 phase 在 A 档最大且最干净的收益。
- **B（无拥塞档）**：base p99 ≈ 15（已经是服务时间下限），phase 反而把 p99 抬到 80–99。物理含义：phase 引入的人为 offset 在没有拥塞需要错峰时，纯粹给每个任务加了 `mean_start_offset` 这一段强制延迟。
- **C（轻拥塞档）**：base p99 ≈ 27–32，phase 压到 15。和 A 同号但量级温和。
- → **判定**：p99 收益与「base 是否已经被拥塞拖延」严格挂钩。phase 在 B 档不仅没有 p99 收益，还是 *p99 损害*；做 STPS 端到端集成时必须先观测 `avg_cong_ratio` 再决定开不开 phase。

### 7.4.4 phase-shift 的 Pareto 关系图（语义版）

把 base `avg_cong_ratio` 作横轴，phase 对各指标的方向写成函数关系：

```
base_cong → 0          base_cong ≈ 0.1        base_cong ≈ 0.26
(B 工作点)              (C 工作点)              (A 工作点)
─────────────          ─────────────          ─────────────
Δcong   = 0 (no-op)     Δcong   ≈ -0.1          Δcong   ≈ -0.26   ← 单调收益
ΔCV     < 0 (改善)      ΔCV     < 0 (改善)      ΔCV     > 0 (恶化) ← 反号
Δp99    > 0 (恶化)      Δp99    < 0 (改善小)    Δp99    < 0 (改善大) ← 反号
Δreject ≈ 0             Δreject ≈ +0.15         Δreject ≈ +0.20   ← 副作用
Δtp     ≈ -7%           Δtp     ≈ -15%          Δtp     ≈ -7%     ← U 型
```

从这个表能读出三件事：

- **`Δcong` 单调随 base_cong 缩放**：phase 的拥塞缓解是直接、可预测、与 cap binding 强度成正比的物理量。
- **`ΔCV` 与 `Δp99` 在 A 档同时反号**：A 档下「想要拥塞缓解就得忍受 CV 升高」是 phase 自带的强制 trade-off。这是 §6 STPS-spatial+phase 在 D=16 上异常的根本原因——STPS-spatial 已经把空间均衡做好，phase 在 A 档上的「CV 收益方向」直接就是负的，叠上去等于反向优化。
- **`Δtp` 的 U 型**：B、A 两档 tp 损失 ~7%，C 档损失 15%。原因：A 档下 base 已经在被拥塞拖慢，phase 错峰后 throughput 反弹补偿了 offset 引入的延迟；C 档 base 没拥塞拖慢，phase offset 直接是净损失。所以 phase 在「中间 cap binding」时 throughput 最差。

### 7.4.5 STPS-spatial + phase 在三档下的特殊性

非-STPS 基线在三档的 phase 方向至少有一档 (A) 上 `Δcong` 干净归零；STPS-spatial 不一样。

| 工作点 | base cong | phase cong | Δcong |
|---|---:|---:|---:|
| A 拥塞 | 0.262 | 0.158 | -0.104 |
| B 无拥塞 | 0.000 | 0.000 | +0.000 |
| C 轻拥塞 | 0.086 | 0.080 | -0.007 |

- A 档 STPS-spatial 是唯一在 phase ON 之后 `avg_cong_ratio` 仍有 ~0.16 残留的算法（其它基线 = 0）。
- C 档 STPS-spatial 残留 0.08（其它基线 = 0）。
- B 档 base 0，phase 0，no-op，正常。
- 物理解释：STPS-spatial 的 hotspot split 已经把高 centrality population 撒到多张卡上，每张卡上残留的 demand 时间序列波动比静态空间策略均匀；phase 的 cross-correlation kernel 在「已经被空间策略削过峰」的 demand 上找不到大幅错峰窗口，所以只能消减部分 cong，剩下的同步残量就直接打到 cap。这一档的 STPS-spatial+phase 是 §6 异常的根因，**§9 的 `D_MAX=2` 是这个工作点上最自然的修补**（搜索面小=不会把已切好的 spatial 解打乱）。

### 7.4.6 对 STPS 端到端集成的判定

把三档发现合并成可执行的设计规则：

1. **`avg_cong_ratio` 是 phase 的开关变量**：检测到 `base avg_cong_ratio < 0.05`（B 档量级）时关掉 phase，避免 p99 反向恶化与 throughput 净损失。
2. **STPS-spatial + phase 在 A 档需要小 `D_MAX`**：§9.5 的 `D_MAX=2` 是 STPS+phase 的最优工作点；`D_MAX=16` 在 A 档导致 §6 异常。
3. **不要把 `card_cv` 当 phase 评价指标**：A 档 phase ON 后 `card_cv` 必然升高，并不代表 phase 失败。phase 在 A 档的有效指标是 `Δcong` 和 `Δp99`。
4. **`reject_rate_bw` 的 ~0.20 是 A 档非-STPS 基线 phase 的固有副作用**：phase 把 demand 错峰后部分任务被推到 `start_offset > 0`，那部分 demand 在 offset window 内仍可能撞到 cap 被拒。STPS-spatial+phase 因为留了 cong 残量、没把所有 demand 推走，反而 `reject_rate_bw` 是 0（A、C 两档），这是另一个支撑「STPS+phase D_MAX=2」配置的角度。

**4 卡已确认的结论：**

1. **Step B 在 `BW_MAX=9e5/D=16` 工作点的物理收益是把 NoC 拥塞压到 0、把 p99 从 ~130 压到 ~30**（非-STPS 基线，原始 p99 口径）。这就是 phase-shift 的核心物理价值。
2. **代价不是「免费」**：`card_cv` 上升 12–50%，`reject_rate_bw` 上升到 ~0.20，吞吐下降 4–12%（剔除冷启动后 3–11%）。
3. **STPS + phase 在 `D=16` 是 Pareto 反向**：p99 反向飙升 68–82%。`D_MAX=2` 是 STPS 默认更合理的取值。
4. **冷启动剔除后吞吐损失收窄 1–2 个百分点**：说明旧口径夸大了 Step B 的吞吐成本约 1/4。

**下一步：**

- 把 STPS 默认 `D_MAX` 改为 2，并在 STPS 论文/STPS 主实验里同步更新数值；4 个静态基线 + phase 保持 `D_MAX=16`。
- 跑 `D_MAX ∈ {2, 4, 8, 16}` 的 Pareto sweep（4 卡），找 CV-vs-p99 拐点。
- 在 `BW_MAX=5e5`（demand p50 以下）的重拥塞档重复 4 卡的对比，确认 §0 的结论在更极端拥塞下是否仍成立。
- 找到能稳定跑完 16 卡 + D=16 + STPS 的环境（32 GB 节点 / streaming），补上 16 卡 Table Q2-2-16 等。

**关键结论：**

1. **phase-shift 的「物理收益方向」由 `BW_MAX` 决定**——它要么压拥塞 + 抬 CV（拥塞档），要么压 CV + 抬尾延迟（无拥塞档）。没有同时占满两边的工作点：cap binding 与否决定 phase 是消化拥塞还是错峰 demand。
2. **无拥塞档下 phase-shift 是「平 CV」的合理模块**：5 个 algo × 2 arrival 全部 CV 改善 12–27%、JFI 改善 +0.05–0.13、LIF 改善 8–17%，**包括 STPS-spatial+phase 这一对**（与拥塞档 §6 异常完全相反）。代价是 p99/avg_delay 上升（因为 baseline 已经几乎无延迟）。
3. **拥塞档下 phase-shift 是「消化 NoC 拥塞 + 压尾延迟」的模块**：5 个非-STPS algo 的 p99 从 ~130 压到 ~30（−75%），avg_cong → 0，但同时把 baseline 已经很低的 CV 拉高（cap-clipped 副作用）。
4. **`Throughput (excl. cs)` 的下降量级在两档下接近一致**（3–11%），说明 ≤10% 的吞吐成本是 phase 模块自身的固有代价，与 `BW_MAX` 无关；剩余被「冷启动算进等待时间」造成的 1–2% 计量假象在两档下也基本一致。
5. **STPS-spatial+phase 在无拥塞档（5e6）+ `D=16` 下不再是 Pareto 反向**：CV / JFI / LIF / Max/Min 全部改进，吞吐损失也是 5 个 algo 中最低（2.8%/8.3%）；这意味着 [§6](#6-stps--phase--d_max16-异常分析) 的「STPS 必须降 D=2」的结论 **只针对拥塞档**。STPS 默认 `D_MAX=2` 在无拥塞档下是否仍最优，需要单独跑 `D ∈ {2, 4, 8, 16} × BW ∈ {9e5, 5e6}` 的 2D Pareto sweep 才能确定。

**关键结论：**

1. **「phase 是否抬 CV」由 `cap binding` 强弱决定，与 `tasks` 量级强相关：** §2 `tasks=800` 时 `BW=9e5` cap 已 binding（avg_cong ~0.26），phase 错峰把任务进一步挤到 cap 边界，反而拉高 CV；§10 `tasks=400` 时 cap binding 弱（avg_cong ~0.08–0.10），phase 错峰直接降低 CV、抬高 JFI/LIF（10/10 行方向一致正向）。这与 [§9](#9-bw_max5e6-对比实验cap-binding-与-phase-shift-收益方向的最终验证) 中「`BW=5e6` 取消 cap binding 也能让 phase 正向」的结论 **吻合**：只要 demand 不撞到 cap，phase 都是平 CV 的合理模块。
2. **吞吐成本在轻拥塞档反而 *更高*（−14.2 ~ −16.4% vs §2 的 −4 ~ −12%）：** `steps=512 / tasks=400` 时已经有大量任务因为 `start_offset` 延迟把完成推到 steady-window 之外，从而被吞吐分母拒绝；这一现象与 §9 (`BW=5e6 / tasks=800`) 中 `−2.8% ~ −10.6%` 的吞吐损失对比说明：**phase 的吞吐成本是 `D / steady_window` 的函数，与 `tasks` 量级无关、与 `BW_MAX` 无关，只跟 `D_MAX` 与 `steps` 比例相关**。
3. **`STPS-spatial+phase` 异常在 `tasks=400` 下仍然存在：** Poisson p99 +20%、Bursty p99 +19.6%、两种 arrival 下 `time_lif` 均上升、`avg_delay` 上升 43–55%。这说明 [§6](#6-stps--phase--d_max16-异常分析) 的「STPS 必须降 `D=2`」结论 **不仅与 `BW_MAX` 有关，也与 `tasks` 量级有关**：只要 `D_MAX=16` 给 STPS 的相位搜索过大，就会出现 hotspot-split 与 phase 错峰互相抵消的现象。结合 §9 的 STPS+phase 正向结果，可以确认：**STPS 在 `D_MAX=16 + cap binding 弱（BW=5e6 或 tasks=400）+ avg_cong ≈ 0` 三个条件中至少有 *两个* 成立时才回归正向**，单一变量松绑（仅 `tasks=400`）不足以解除 STPS 行的 Pareto 反向。
4. **`reject_rate_bw` 是 cap binding 的唯一直接物理指标：** 三个工作点对照：§1 `tasks=800/BW=9e5` 拒绝率 0.18–0.22；§10 `tasks=400/BW=9e5` 拒绝率 0.14–0.16；§9 `tasks=800/BW=5e6` 拒绝率 0。`tasks` 减半（demand 减半）只能把拒绝率从 0.20 压到 0.15，而 `BW` 扩大 5.5×（cap 抬高 5.5×）能完全消除拒绝。这定量地说明：在 `BW=9e5` 这一档，**`tasks` 量级与 `BW_MAX` 量级在 cap binding 上不是对称的**——前者只能线性减小拒绝，后者可以直接解除拒绝。
5. **`CV mean/median/p95/std` 四列在 `tasks=400` 下的分散度（`std` 列）整体小于 §1：** baseline std ~0.006–0.028，phase-on std ~0.011–0.033；§1 同列 baseline std ~0.014–0.050。说明 `tasks=400` 工作点的 CV 比 `tasks=800` 工作点的 **5-seed 稳定性更好**（数据更可信），可作为「phase-shift 在轻拥塞档下普遍正向」的强证据。

数据齐全；§10 的方向与 §9 互为印证：**「cap binding 弱」是 phase-shift 在 4 卡上稳定正向降 CV 的充分条件**，不论是用降低 `tasks`（demand 减半）还是抬高 `BW_MAX`（cap 抬高）来实现。
