# Q2 实验结果：Phase-Shift（Step B）净收益 — 4 卡，**FIFO NoC 注入队列模型**

> NoC 服务模型已从「比例缩放削峰」换成 **有界 FIFO 注入队列**（commit b3915c7）：每张卡按 ≤ `bw_cap` 排空一条深度 `queue_depth_factor × cap`（默认 8）的 FIFO，满则背压（不丢任务）；拥塞按 backlog 重算 `cong = backlog / (served + backlog)`，`cap=None` 退化为无队列。三档已用新模型重跑（`script/q2_run.py` 默认 `queue_depth_factor=8`）。吞吐用剔除冷启动口径 `throughput_excl_cold = N/(T − mean_cold_start)`；p99 用原始口径 `p99(completion − arrival)` 不扣冷启动；avg delay 提供 excl. cs 列。
>
> **范围说明：** 本轮以 4 卡完整跑通三档；16 卡因 D=16/STPS 内存峰值在小内存主机跑不完暂缓（§7 备注）。

## 0. 结论先行

新 FIFO 队列模型下三档（A 拥塞 `bw9e5_t800` / B 无拥塞 `bw5e6_t800` / C 轻拥塞 `bw9e5_t400`，均 `D_MAX=16`）的 phase-shift 净效应，与旧比例缩放模型相比**画风一致化**：

**1. phase-shift 在三档都降低 CV（仅 STPS-spatial+A 例外）。** 非-STPS 基线 `card_cv gain`：A 档 +12.8%–23.2%，B 档 +12.5%–27.2%，C 档 +21.1%–39.0%，全部正向（phase 平衡负载）。旧模型 A 档「phase 抬 CV」的反向已消失：FIFO 队列不削平 baseline，CV 方向不再被 cap-clipping 翻转。

**2. 拥塞被消化，但只在 cap binding 时。** A/C 两档非-STPS 基线 `avg_cong_ratio` 从 0.11–0.14 / 0.04–0.06 压到 0；B 档本就 0，phase 无拥塞可解。

**3. p99 三档都上升（offset 代价），绝对值由 baseline 拥塞决定。** A：15→30（+~100%）；C：15→16–30；B：15→80–108（+440–625%，无拥塞档最贵）。avg_delay (excl. cs) 几乎不变。p99 收益不再存在——FIFO 队列下 baseline 不深排队，phase 引入的 cold-start 净增尾延迟。

**4. STPS-spatial+phase 仅在 A 档 `D=16` 异常。** A 档 CV +1.6%/−14.4%、p99 82–100 tick；B/C 档 CV 转正（+5.7%–19.8%）。STPS 行吞吐几乎零损失（reject=0）。

**使用建议：**

- **拥塞 + 均衡敏感**：A/C 档 phase 降 CV、清拥塞，代价是吞吐 14–24% + reject 0.14–0.22。
- **无拥塞**：B 档 phase 仍降 CV 但 p99 暴涨 5–6×，需按 `avg_cong_ratio<0.05` 关闭。
- **STPS 默认 `D_MAX=2`** 而非 16，避免 A 档 stps-spatial 上 p99 拖到 82–100 tick。

- **`reject_rate_bw` 上升到 0.18–0.22**：非-STPS 基线 phase=on 后，约 1/5 的任务在 `D_MAX=16` 范围内仍找不到 ≤ cap 的 slot 而被拒。

## 6. STPS + Phase + `D_MAX=16` 异常分析

§2/§3 多次提到 **stps-spatial+phase（即默认 STPS 全模型）在 `D=16` 下 p99 反向飙升**。复现旧 Q2 §9.3 的结论，原因可拆为两点：

**stps-spatial 自身已经把同步热点做空间拆分**：再用 `D=16` 大窗口去搜索时把零散需求推到稳态窗尾部，p99 升到 82–100 tick、CV 反升；非-STPS 基线 baseline 仍高度同步，需要大窗口才能压平，方向正好相反。

**结论：STPS 端到端默认应使用 `D_MAX=2`**。本轮 `BW=9e5/D=2`（regime C，§7.1 主表）下 stps 的 CV 0.62/0.71、p99 17 tick、吞吐零损失，全面优于 `D=16` 的 stps-spatial+phase（A 档 CV 0.43/0.45、p99 82/100 tick）。9e5/D=16 数据再一次证实 D=16 不适合 STPS。

非-STPS 基线（RR / BestFit / DRF / P2C）+ phase + D=16 没有这个问题，因为它们的 baseline demand 仍然高度同步，phase 用大窗口才能压平；这一点和 STPS 模块的需求正好相反。

## 7. 16 卡版本备注

16 卡（`cards=16, tasks=3200`）版本本轮未跑完。原因：

- 单个 16 卡 + `D_MAX=16` STPS-phase worker 的 RSS 峰值 4–6 GB；在 15 GB 主机上跑 2 worker 时一旦同步进入 STPS 阶段就接近 OOM。
- 单 worker 时 STPS-spatial / STPS-phase 单 run ~10 min，60 个剩余 run ≈ 5–6 h，本轮没有跑完的不间断窗口。
- `script/q2_run.py` 已加入 `multiprocessing.Pool(maxtasksperchild=1)` + 增量 CSV 写盘 + resume 机制，下次有 32 GB 节点或更长不间断窗口时可以无缝接着跑：

  ```bash
  /usr/bin/python3 script/q2_run.py main16 \
      --bw-max 9e5 --d-max 16 --name main16_bw9e5_d16 --workers 1
  ```

  会自动跳过 `data/q2/main16_bw9e5_d16_raw.csv` 中已有的行（poisson RR/BestFit/DRF/P2C × 5 seed × 2 phase + 5 STPS-spatial 已完成，共 45 行）。

按 4 卡的物理机理推断，16 卡上 Step B 的方向预期：congestion 缓解、p99 显著下降、card_cv 因卡多基线更低而 ΔCV 可能更温和、STPS+phase 仍然在 `D=16` 异常。等 16 卡跑完后会单独补充 Table Q2-2-16 / Q2-3-16 / Q2-4-16 / Q2-5-16 / Q2-6-16 / Q2-7-16 等。

### 7.4.3 三个核心发现

**发现 1 — phase-shift 是「拥塞缓解器 + 负载均衡器」**

- 4 个非-STPS 基线在 A、C 两档（cap binding 存在）下 `avg_cong_ratio` 被 phase 压到 0；B 档本来就 0，无拥塞可缓解。
- FIFO 队列下 phase 同时降 CV：拥塞缓解只在 cap binding 时，CV 改善在三档都有。

**发现 2 — `card_cv` 三档都改善（仅 STPS+A 例外）**

- A/B/C 三档非-STPS 基线 CV 都下降 12–39%；FIFO 队列不削平 baseline，A 档不再出现「phase 抬 CV」。
- 唯一反向是 STPS-spatial+A（−9%）：已切空间再叠 D=16 phase 反而过度去同步。
- → **判定**：phase 是稳定的 CV 降低器，CV 在三档都能用来评价 phase（除 STPS+A）。

**发现 3 — `p99` 三档都上升，B 档最贵**

- 三档 baseline p99 均 ~15 tick（FIFO+backpressure 无深排队）；phase 加 `mean_offset` 后净增尾延迟。
- A：15→30；C：15→16–30；B：15→88–108（无拥塞可重叠 offset，最贵）；STPS+A：15→91。
- → **判定**：phase 没有 p99 收益；做端到端集成要在均衡收益与尾延迟代价间权衡。

### 7.4.4 phase-shift 的 Pareto 关系图（语义版）

把 base `avg_cong_ratio` 作横轴，phase 对各指标的方向写成函数关系：

```
base_cong → 0          base_cong ≈ 0.05       base_cong ≈ 0.13
(B 工作点)              (C 工作点)              (A 工作点)
─────────────          ─────────────          ─────────────
Δcong   = 0 (no-op)     Δcong   ≈ -0.05         Δcong   ≈ -0.13   ← 单调收益
ΔCV     > 0 (改善)      ΔCV     > 0 (改善大)    ΔCV     > 0 (改善)  ← 同号
Δp99    ≫ 0 (大恶化)    Δp99    > 0 (小恶化)    Δp99    > 0 (小恶化) ← 同号
Δreject ≈ 0             Δreject ≈ +0.15         Δreject ≈ +0.20   ← 副作用
Δtp     ≈ -7%           Δtp     ≈ -15%          Δtp     ≈ -21%    ← 随 cap 单调
```

读出三件事：

- **`Δcong` 单调随 base_cong 缩放**：phase 的拥塞缓解直接、可预测、与 cap binding 成正比。
- **`ΔCV` 三档全正**：FIFO 队列下 phase 降 CV 与 cap binding 无关；唯一反向是 STPS+A。
- **`Δtp` 随 cap binding 单调**：B −7% → C −15% → A −21%，A 因 reject 0.20 成本最高。

### 7.4.5 STPS-spatial + phase 在三档下的特殊性

| 工作点 | base cong | phase cong | Δcong |
|---|---:|---:|---:|
| A 拥塞 | 0.120 | 0.068 | -0.052 |
| B 无拥塞 | 0.000 | 0.000 | +0.000 |
| C 轻拥塞 | 0.044 | 0.037 | -0.007 |

- A/C 档 STPS-spatial phase ON 后 cong 仍有残留（0.068/0.037），其它基线 = 0。
- 物理解释：hotspot split 已削过峰，phase 只能消减部分 cong；A 档 D=16 还把 CV 推高、p99 升到 91 tick——`D_MAX=2` 是该工作点的修补。

### 7.4.6 对 STPS 端到端集成的判定

1. **STPS-spatial + phase 在 A 档需要小 `D_MAX`**：`D_MAX=2`（§7.1）是 STPS+phase 最优，CV 0.62/0.71、p99 17；`D_MAX=16` 在 A 档导致 §6 异常。
2. **p99 是 phase 的代价开关**：base p99 已是 15 tick 下限，需评估能承受的尾延迟，B 档（无拥塞）最不划算。
3. **`reject_rate_bw` ~0.20 是 A 档非-STPS 基线的固有副作用**；STPS+phase reject=0（留了 cong 残量），是支撑 `D_MAX=2` 的另一角度。

## 8. 结论与下一步

**4 卡已确认的结论（FIFO 注入队列模型）：**

1. **phase-shift 三档都降 CV**：非-STPS 基线 `card_cv gain` A +13–23% / B +12–27% / C +21–39%，仅 STPS-spatial+A 反向。FIFO 队列不再削平 baseline，旧模型「拥塞档抬 CV」的反向消失。
2. **拥塞缓解只在 cap binding 时**：A/C 把 `avg_cong_ratio` 0.11–0.14 / 0.04–0.06 压到 0；B 本就 0。
3. **代价是吞吐 + 尾延迟**：吞吐降 A 18–23% / C 14–16% / B 3–10%；p99 三档全升（A 15→30、C 15→16–30、B 15→80–108），因 FIFO+backpressure 下 baseline p99 已是 15 tick 下限，offset 净增尾延迟。
4. **STPS + phase 在 `D=16` 仅 A 档异常**：p99 82–100 tick、CV 反向；B/C 转正。`D_MAX=2` 是 STPS 默认更合理取值，且吞吐零损失。

**下一步：**

- STPS 默认 `D_MAX=2`；4 个静态基线 + phase 保持 `D_MAX=16`。
- 跑 `D_MAX ∈ {2, 4, 8, 16}` 的 Pareto sweep（4 卡），找 CV-vs-p99 拐点。
- 在更极端拥塞档（demand p50 以下）复核三档结论是否仍成立。
- 找到 32 GB 节点补上 16 卡 + D=16 + STPS。
