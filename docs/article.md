# Phase-Shift `D_MAX` 敏感性分析 — 轻拥塞档 C（`BW_MAX=9e5`, `tasks=400`）

> 本文档汇总论文使用的 phase-shift 延迟预算（`D_MAX`）扫描实验。工作点固定在 **轻拥塞档 C**（`bw9e5_t400`）：`BW_MAX=9e5`（demand p75，`docs/traffic_calib.md §3`）配合 `tasks=400`（demand 折半），cap binding 较弱（baseline `avg_cong_ratio ≈ 0.08–0.10`，`reject_rate_bw` phase=on 时 ~0.14–0.16）。此档位的工作点定义与原始数据见 `docs/Q2_result.md` §2.3 / §10.1 与 `data/q2/main4_bw9e5_d16_t400_*.csv`。
>
> 本轮把 `D_MAX` 设为 **2 / 4 / 8 / 16** 四组，重点观察 **吞吐量、跨卡负载均衡、p99 延迟、拥塞程度** 随延迟预算的变化。`D_MAX=16` 沿用既有数据；`D_MAX∈{2,4,8}` 为本轮重跑。

## 3. 分析

把四个非-STPS 基线（RR / BestFit / DRF / P2C）和 STPS-spatial 分开看，`D_MAX` 的作用机制完全不同。

### 3.1 拥塞程度：一个开关，与 `D_MAX` 无关

四个非-STPS 基线只要 **打开 phase（任意 `D_MAX`≥2）**，`avg_cong_ratio` 就从 baseline 的 0.08–0.10 直接压到 **0.0000**，`cong.wait` 同步归零。轻拥塞档下 cap binding 较弱，`D_MAX=2` 的搜索窗已足够把同步打到 cap 的需求摊到时间维——**拥塞消化是一个二值开关，不随 `D_MAX` 增大而继续改善**（已经到 0，没有下行空间）。

STPS-spatial 则不同：它本身已在空间维切过载，phase 叠加后 `cong.ratio` 仍残留 0.08 左右，且随 `D_MAX` 增大 **缓慢下行**（poisson 0.0868→0.0798，bursty 0.0879→0.0792）。STPS 的拥塞是空间切分后的残差，phase 在更大预算下能多消化一点，但量级很小。

### 3.2 负载均衡：`D_MAX` 越大越平，但 P2C 反向

非-STPS 基线的跨卡负载均衡（CV↓ / JFI↑ / LIF↓）**随 `D_MAX` 单调改善**，到 `D_MAX=8` 附近见顶：

- **RR**：CV 从 baseline 0.676（poisson）随 `D_MAX` 一路降到 `D=8` 的 **0.4425**（最优），`D=16` 略回升到 0.4835；JFI 在 `D=8` 达到 0.823 峰值。bursty 同形：`D=8` CV=0.5604 为最低。
- **BestFit / DRF**：CV 单调下降（poisson 0.6466→0.4799，bursty 0.6830→0.5525），`D=16` 最优；`Max/Min` 从 baseline 13.5（poisson）压到 5.0 量级。
- **P2C 是反例**：`D_MAX=2` 时 CV **反而恶化**（poisson 0.717→0.832，bursty 0.735→0.875），需要更大的 `D_MAX` 才把 CV 拉回基线以下（poisson 要到 `D=16` 才 0.6565 < 0.717）。P2C 的「两选一」本身已带随机均衡，小预算 phase 把每张卡的 serve window 错开，在 cap-clipped 下反而拉大了跨卡瞬时方差——这与 `docs/Q2_result.md §2.1` 拥塞档观察到的 P2C 对 phase 最敏感一致。

STPS-spatial 的 CV 改善很小（poisson 0.659→0.601，单调但幅度仅 ~9%），`Max/Min` 在 11–13 区间无明显趋势——空间维已经吃掉大部分均衡收益，phase 在残余维度上能动的空间有限。

### 3.3 p99 延迟：与 `D_MAX` 正相关（代价随预算增长）

p99 的方向与负载均衡 **相反**——`D_MAX` 越大，phase 注入的 `start_offset` 越大（`mean offset` 列从 ~0.5 tick 升到 ~4 tick），尾延迟越高：

- 非-STPS 基线：phase 仍是 **净降 p99**（baseline 24–33 → phase-on 15–28），但收益随 `D_MAX` 增大而 **缩水**。以 poisson RR 为例：`D=2` p99=15.0（降 52%），`D=16` p99=16.2（仍降，但 `mean offset` 已涨到 2.32）。P2C/BestFit 在 `D=16` 时 p99 已逼近甚至接近基线（poisson P2C `D=16` p99=25.9 vs baseline 29.0）。
- STPS-spatial：phase 在此档 **始终抬高 p99**（poisson 25.0→30.0，bursty 28.6→34.2），且随 `D_MAX` 单调恶化。这复现了 `docs/Q2_result.md §6` 的 STPS+phase 异常——STPS 已切过空间，phase 在剩余维度找不到更优 slot，只是把 offset 等待累加进尾延迟。**STPS 默认应取 `D_MAX=2`**。

### 3.4 吞吐量：phase 的固定成本 + `D_MAX` 的部分回收

非-STPS 基线打开 phase 后吞吐 **明显下降**（0.76 → 0.59–0.64），且 **随 `D_MAX` 增大而部分回收**：poisson RR `D=2` 0.593 → `D=16` 0.658。机制有二：

1. **`reject_rate_bw` 随 `D_MAX` 下降**（poisson RR 0.228→0.142）：预算越大，越多任务能在 `[0,D_MAX]` 内找到 ≤ cap 的 slot，被拒比例下降，完成数回升。
2. **稳态窗口边界效应**：`tasks=400 / steps=512` 下，被 offset 推迟的任务更容易落到稳态窗（首尾各去 64 tick）之外。`tput(xc)`（剔除冷启动）与 `tput` 几乎一致（差 <0.005），说明吞吐损失 **主因是 cap 拒绝 + 窗口边界，而非冷启动计量**——与 `docs/Q2_result.md §10.6` 的解读一致。

STPS-spatial 吞吐几乎不掉（0.76 全程），因为它 `reject_rate_bw=0`（hotspot-split 改变 `task_size` 后总能放下），phase 仅引入极小 offset。

### 3.5 小结：`D_MAX` 的 Pareto 取舍

| 维度 | `D_MAX` 增大的方向 | 说明 |
|---|---|---|
| 拥塞 `cong.ratio` | 非-STPS 已在 `D=2` 归零，无额外收益；STPS 缓慢下行 | 拥塞消化是开关，不是旋钮 |
| 负载均衡 CV/JFI/LIF | **改善**（RR/BestFit/DRF 单调，`D=8` 附近见顶）| P2C 需大预算才转正 |
| `Max/Min` | **改善**（baseline 8–13 → 3–6）| `D=2` 已拿到大部分 |
| p99 / mean offset | **恶化**（offset 等待随预算线性增长）| phase 的主要代价 |
| 吞吐 | **部分回收**（reject 下降）| 但始终低于 baseline |

**推荐工作点**：

- **非-STPS 基线 + 延迟敏感**：`D_MAX=2`——`cong.ratio` 已归零、p99 降幅最大（~50%）、`Max/Min` 已压到 3–6，代价（offset、吞吐损失）最小。
- **非-STPS 基线 + 均衡敏感**：`D_MAX=8`——CV/JFI 见顶，p99 仍可控（16–22），reject 已降到 ~0.17。`D_MAX=16` 的额外均衡收益很小却显著抬高 p99，不推荐。
- **STPS**：`D_MAX=2`——更大的预算只抬高 p99、不带来均衡或拥塞收益。
